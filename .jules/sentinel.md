## 2026-03-20 - [Vaultwarden Root Privilege Avoidance]
**Vulnerability:** Vaultwarden container was running as root and using a privileged port (80).
**Learning:** Setting `readOnlyRootFilesystem: true` on Vaultwarden containers is risky and will cause crashes unless writable volume mounts (such as `emptyDir`) are explicitly provided for required directories like `/tmp`. To securely harden Vaultwarden's `securityContext`, avoid granting the `NET_BIND_SERVICE` capability. Because the official image defaults to root, you must explicitly define `runAsUser`, `runAsGroup`, and `fsGroup` (e.g., 1000) alongside `runAsNonRoot: true`. Drop all capabilities (`capabilities: drop: ["ALL"]`) and reconfigure the application to use an unprivileged port via the `ROCKET_PORT` environment variable (e.g., "8080").
**Prevention:** Apply the standard security hardening pattern for Kubernetes deployments: pod-level `runAsNonRoot: true`, container-level `allowPrivilegeEscalation: false` and `readOnlyRootFilesystem: true`, accompanied by an `emptyDir` volume mounted at `/tmp`. Configure apps to listen on unprivileged ports.
