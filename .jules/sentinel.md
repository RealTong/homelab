## 2025-03-05 - Vaultwarden Security Hardening Nuances
**Vulnerability:** Vaultwarden was running with excessive privileges (root, writable root filesystem).
**Learning:** Hardening Vaultwarden's `securityContext` requires specific handling because it binds to port 80 by default. When enforcing `runAsNonRoot: true` and `readOnlyRootFilesystem: true`, a writable volume (like `emptyDir`) must be mounted to `/tmp` for temporary files. Furthermore, to avoid granting `NET_BIND_SERVICE` capability to a non-root process, the application port should be changed to an unprivileged port (e.g., 8080) using the `ROCKET_PORT` environment variable, and the container and service ports updated accordingly. Also, dropping all capabilities (`capabilities: drop: ["ALL"]`) and setting `fsGroup`, `runAsUser`, and `runAsGroup` is a more secure and robust approach.
**Prevention:** Apply the standard security hardening pattern (pod-level `runAsNonRoot: true`, container-level `allowPrivilegeEscalation: false`, `readOnlyRootFilesystem: true`, and dropping all capabilities) to all Kubernetes deployments. Remember to provide writable mounts (e.g., `/tmp`), set explicit user/group IDs, and remap privileged ports to unprivileged ones.
