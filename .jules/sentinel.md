## 2024-03-08 - [HIGH] Vaultwarden Root Privileges
**Vulnerability:** Vaultwarden containers in the homelab run as root by default with a writable filesystem.
**Learning:** Setting `readOnlyRootFilesystem: true` on Vaultwarden containers causes them to crash without explicitly mounting a writable `emptyDir` volume to `/tmp`. Furthermore, adding `runAsNonRoot: true` requires specifying `runAsUser`, `runAsGroup`, and `fsGroup` to run successfully (e.g., 1000) since the image defaults to root. Finally, it requires configuring the container to run on an unprivileged port via the `ROCKET_PORT` environment variable (e.g., "8080").
**Prevention:** In Kubernetes deployment manifests for applications like Vaultwarden, explicitly set Pod-level security context details (`runAsUser`/`runAsGroup`/`fsGroup`), and when enabling a read-only root filesystem, identify and provide emptyDir mounts for required writable directories like `/tmp`. Also bind to non-privileged ports when running as non-root without `NET_BIND_SERVICE`.
